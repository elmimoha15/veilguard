# Auto-Scanning

Veilguard scans automatically. You never type a command. You never open a dashboard.

---

## How it works

Veilguard uses a **security rules file** in your project root. This file tells your AI agent exactly which Veilguard tool to call, and when. You download it once during setup (see the [per-IDE installation guides](/docs/installation/cursor)).

| IDE | Rules file |
|-----|-----------|
| Windsurf | `.windsurfrules` |
| Cursor | `.cursorrules` |
| VS Code | `.cursorrules` |
| Claude Code | `CLAUDE.md` |
| Antigravity | `.antigravityrules` |

Your AI agent reads these rules on every prompt and follows them automatically.

---

## Trigger table

| When this happens | Veilguard calls |
|-------------------|----------------|
| You create or modify any `.ts`, `.tsx`, `.js`, `.jsx`, `.sql`, `.json`, `.env`, or config file | `scan_secrets` |
| You create or modify an API route, endpoint, webhook handler, or server action | `scan_webhooks` + `scan_injection` |
| You create or modify a Supabase migration, SQL file, or RLS policy | `check_supabase_rls` |
| You create or modify Firebase rules | `check_firebase` |
| You modify `package.json` or add a new dependency | `check_supply_chain` + `scan_dependencies` |
| You modify `.env`, `.gitignore`, `next.config.js`, or any config file | `check_env` |
| You say "deploy", "push to prod", "go live", "ship it", "launch", or "publish" | `full_audit` (Pro) or all free scanners in sequence |
| The first time you work on a new project | `scan_secrets` + `check_env` |
| You ask anything like "is my app secure?", "audit my code", "what's my security grade?" | `full_audit` |

## The silent scan rule

**If a scan is clean: total silence.** Your AI says nothing. You won't even know a scan ran.

**If something is found:** your AI mentions it in plain English at the right moment â€” critical issues at the top of the response, warnings at the bottom. No jargon, no scanner names, no file paths in the summary (unless you ask for them).

### Example: AI catches a hardcoded key

You write a file with `const STRIPE_KEY = "sk_live_..."`. Your AI agent:

1. Calls `scan_secrets` silently (you don't see this step)
2. Returns a finding
3. Reports it as: *"ðŸ”´ Your Stripe live key is hardcoded in source code â€” anyone who sees your repo can steal it and charge your customers."*
4. Offers to fix it

### Example: Clean scan

You write a new API route. Your AI agent:

1. Calls `scan_webhooks` and `scan_injection` silently
2. Both return clean
3. Says nothing about security â€” proceeds with your request as normal

## Claude Code: the PostToolUse hook

Claude Code gets an extra, deterministic layer of protection beyond what the AI agent decides to do â€” a real hook that fires every time the agent edits or writes a file.

`veilguard init` adds a `PostToolUse` hook to `.claude/settings.local.json` (project-local, gitignored):

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "npx -y --package=veilguard veilguard-cli scan-hook"
          }
        ]
      }
    ]
  }
}
```

This means:
- Every time the AI **edits or writes** a file, `scan-hook` scans **just that file** and routes the right scanners by file type (secrets/injection always; webhooks + RLS for source files; supply-chain for `package.json`; etc.).
- If the file is **clean**, the hook exits silently â€” nothing is shown.
- If it finds a vulnerability, the finding is injected back into the conversation and the agent reports it in plain English and offers a fix.

The hook fires on the agent's edits, not on files you save by hand in another editor. Because `PostToolUse` runs after the write, it never blocks your edit.

## Security / audit keyword triggers

When you use any phrase that implies you want a full security assessment, your AI agent **must** call `full_audit` instead of improvising its own checklist. This is enforced by the rules file.

Phrases that trigger a full audit (and semantic equivalents):

- "full audit" / "security audit" / "audit my app"
- "is my app secure?" / "is my app safe?"
- "what's my security grade?" / "grade my app"
- "ready to deploy?" / "can I deploy?" / "safe to launch?"
- "scan everything" / "scan my whole project"
- "check my security"

The AI calls `full_audit` silently and returns the report in plain English. It never substitutes its own opinion for a real scan.
